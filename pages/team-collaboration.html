チーム開発の基本

個人開発とチーム開発の最大の違いは「コミュニケーション」です。コードを書く技術だけでなく、チームメンバーと協力する方法を学びましょう。

💡 このページで学べること：
・GitHubを使ったチーム開発の流れ
・プルリクエストの作成とレビュー
・効果的なコミュニケーション方法
・チーム開発のベストプラクティス

ブランチ戦略

GitHub Flow（推奨）

シンプルで初心者にも分かりやすいワークフローです。

基本の流れ

  1. mainブランチは常にデプロイ可能な状態を保つ
  2. 新しい機能やバグ修正は、mainから新しいブランチを作成
  3. 作業が完了したらプルリクエストを作成
  4. レビューを受けてマージ
  5. mainブランチにマージしたら自動デプロイ

ブランチ名の付け方

# 機能追加
feature/add-search-function
feature/user-authentication

# バグ修正
fix/login-button-not-working
fix/header-layout-issue

# ドキュメント
docs/update-readme
docs/add-api-documentation

実際の操作手順

1. 最新の状態を取得

git checkout main
git pull origin main

2. 新しいブランチを作成

git checkout -b feature/add-contact-form

3. 作業してコミット

git add .
git commit -m "feat: お問い合わせフォームを追加"

4. リモートにプッシュ

git push origin feature/add-contact-form

5. GitHub上でプルリクエストを作成

プルリクエストの作成

良いプルリクエストの書き方

タイトル

  • 何をしたかを明確に
  • プレフィックスを付ける（feat:, fix:, docs:など）
✅ Good
feat: お問い合わせフォームにバリデーション機能を追加
fix: ログインボタンが押せない問題を修正

❌ NG
フォーム修正
バグ直した

説明文のテンプレート

## 概要
お問い合わせフォームに入力チェック機能を追加しました。

## 変更内容
- 必須項目のバリデーション追加
- メールアドレスの形式チェック
- エラーメッセージの表示機能

## スクリーンショット
（必要に応じて画像を添付）

## テスト
- [ ] 必須項目を空白で送信できないことを確認
- [ ] 無効なメールアドレスでエラーが出ることを確認
- [ ] 正しい入力で送信できることを確認
- [ ] スマホで正常に動作することを確認

## 関連Issue
Closes #123

プルリクエスト作成手順

  1. GitHubのリポジトリページを開く
  2. 「Pull requests」タブをクリック
  3. 「New pull request」をクリック
  4. base: main ← compare: あなたのブランチ を選択
  5. タイトルと説明文を記入
  6. レビュワーを指定（Reviewers）
  7. 「Create pull request」をクリック

コードレビュー

レビューを受ける側

心構え

  • レビューは「コードの品質を高めるため」のもの
  • 指摘は個人攻撃ではない
  • 分からない指摘は質問してOK
  • 納得できる説明があれば修正する

レビュー依頼時のポイント

  • 変更が大きすぎないか（1PR = 1機能が理想）
  • 説明が十分か
  • 自分でテストしたか
  • コンフリクトがないか

レビューをする側

レビューの観点

  • 動作： 意図した通りに動くか
  • 可読性： コードが読みやすいか
  • 保守性： 後で修正しやすいか
  • セキュリティ： 脆弱性はないか
  • パフォーマンス： 無駄な処理はないか

コメントの書き方

❌ NG（攻撃的）
「この書き方は間違ってる」
「なんでこんなコード書いたの？」

✅ Good（建設的）
「〇〇の理由で、△△のように書くとより良いと思います」
「もし〇〇の場合、この処理は問題ありませんか？」
「提案：〇〇のように書くとシンプルになりそうです」

承認の種類

  • Approve： 問題なし、マージしてOK
  • Request changes： 修正が必要
  • Comment： コメントだけ（判断は任せる）

コミットメッセージの書き方

良いコミットメッセージの条件

  • 何をしたかが一目で分かる
  • なぜその変更をしたかが分かる
  • 後から見返したときに理解できる

フォーマット

type: 変更内容の要約（50文字以内）

詳細な説明（必要に応じて）
- なぜこの変更が必要だったか
- どのように実装したか
- 注意点があれば記載

type の種類

  • feat: 新機能
  • fix: バグ修正
  • docs: ドキュメント
  • style: コードフォーマット
  • refactor: リファクタリング
  • test: テスト追加・修正
  • chore: その他（ビルド、設定ファイルなど）

例

✅ Good
feat: お問い合わせフォームにバリデーション機能を追加

必須項目（名前、メール、内容）のチェック機能を追加。
空白で送信できないようにし、メールアドレスの形式も検証する。

fix: ログインボタンが押せない問題を修正

z-indexの競合により、モーダルの背後に隠れていたため、
z-index値を調整して解決。

❌ NG
フォーム修正
バグ直した
いろいろ変更

コミュニケーション

GitHubでのコミュニケーション

Issue（課題管理）

  • バグ報告、機能リクエスト、質問などに使う
  • テンプレートを用意しておくと便利
  • ラベルを活用（bug, enhancement, question など）

Discussion（議論）

  • 設計の相談、アイデアの共有に使う
  • Issueにするほどではない話題に

プルリクエストでのやり取り

  • コードの特定行にコメントできる
  • 会話をResolveして解決済みにする
  • 絵文字を活用して雰囲気を和やかに👍

チャットツール

Slack / Discord

  • 技術的な質問： 専用チャンネルを作る
  • 進捗報告： 毎日決まった時間に報告
  • 雑談： チーム内のコミュニケーション活性化

コミュニケーションのコツ

  • 困ったらすぐに質問（抱え込まない）
  • 質問するときは「何を試したか」も伝える
  • リアクション絵文字で反応を示す
  • 非同期コミュニケーションを意識（すぐ返信を期待しない）

チーム開発のベストプラクティス

✅ すべきこと

  1. こまめにpull - 作業前に必ず最新の状態に
  2. 小さくcommit - 1コミット = 1つの論理的な変更
  3. 早めにpush - 1日の終わりには必ずpush
  4. PRは小さく - 大きな変更は分割する
  5. レビューは迅速に - 24時間以内に反応する
  6. テストを書く - 自動テストでバグを防ぐ
  7. ドキュメントを更新 - コードと同時に更新

❌ してはいけないこと

  1. mainブランチに直接commit - 必ずブランチを作る
  2. force push - チームメンバーの作業を壊す可能性
  3. 巨大なPR - レビューしづらい、バグが潜みやすい
  4. レビューなしでマージ - 最低1人はレビューを受ける
  5. テストしないでpush - 必ず動作確認してから
  6. コンフリクトを放置 - 見つけたらすぐに解決

トラブル時の対処

コンフリクトが発生したら

1. 最新のmainを取得
git checkout main
git pull origin main

2. 自分のブランチにマージ
git checkout your-branch
git merge main

3. コンフリクトを解決
（ファイルを編集）

4. 解決をコミット
git add .
git commit -m "Merge main into your-branch"
git push

間違えてcommitしてしまったら

# pushしていない場合
git reset HEAD~1  # 1つ前のコミットに戻る

# pushしてしまった場合
# チームに相談してから対処

Cursorでのチーム開発

.cursorrulesの共有

プロジェクトのルールをCursorに教えることで、チーム全体で一貫した開発ができます。

# .cursorrules（プロジェクトルートに配置）
# コーディング規約
- 変数名はcamelCase
- 関数名は動詞で始める
- コメントは日本語で書く

# コミットメッセージ
- Conventional Commitsに従う
- 日本語で書く

# PRの作成
- 変更は200行以内に抑える
- テストコードも一緒にcommit

このファイルをGitにcommitすることで、チーム全員が同じ設定を共有できます。

まとめ

🤝 チーム開発で大切なこと

  • コミュニケーション： 困ったら相談、進捗は共有
  • 尊重： お互いのコードと意見を尊重
  • 責任： 自分のコードに責任を持つ
  • 学び： レビューは学びの機会
  • 改善： プロセスを継続的に改善
💡 最後に：
完璧なチームはありません。失敗しながら学び、少しずつ改善していきましょう。
