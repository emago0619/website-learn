このページについて

「Gitって名前は聞いたことあるけど、実際何なの？」「コミットとプッシュって何が違うの？」 そんな疑問に、初心者のタロウさんとベテランエンジニアの先輩の対話を通じて答えていきます。

登場人物：
タロウ：プログラミング初心者。Webサイトを作りたいと思っている
先輩：優しく教えてくれるベテランエンジニア

第1話：「Gitって何？バージョン管理って何？」

タロウ：先輩、Gitって何ですか？よく聞くんですけど、正直よくわからなくて...

先輩：いい質問だね！Gitは「ファイルの変更履歴を記録するツール」なんだ。 例えば、タロウくんがレポートを書いてる時のことを思い出してみて。

タロウ：レポート...ですか？

先輩：こんなファイル名、作ったことない？

  • レポート.docx
  • レポート_修正版.docx
  • レポート_修正版2.docx
  • レポート_最終版.docx
  • レポート_最終版_ほんとに最終.docx

タロウ：あー！あります！めっちゃあります！（笑）

先輩：でしょ？これって、「前のバージョンに戻せるように」残してるんだよね。 でもファイルがどんどん増えて、どれが最新かわからなくなる...

タロウ：そうなんですよ！で、結局どれが最新かわからなくなって...

先輩：それを解決するのがGit！Gitを使うと：

  • ✅ ファイルは1つだけ（レポート.docx）
  • ✅ でも、変更履歴は全部記録される
  • ✅ 過去のバージョンにいつでも戻せる
  • ✅ 「いつ」「誰が」「何を」変更したかがわかる

タロウ：すごい！それなら「最終版_ほんとに最終_今度こそ最終」みたいなファイル作らなくていいんですね！

先輩：その通り！（笑）これがバージョン管理なんだ。

第2話：「GitとGitHubは違うの？」

タロウ：あれ、でもGitHubっていうのもよく聞くんですけど...GitとGitHubって違うんですか？

先輩：おっ、いいところに気づいたね！これ、みんな最初混乱するんだ。 簡単に言うと：

  • Git = 自分のパソコンで使うツール（ソフトウェア）
  • GitHub = インターネット上の保管場所（サービス）

タロウ：うーん...まだちょっと...

先輩：じゃあ、写真で例えてみようか。

  • Git = スマホのカメラアプリ（写真を撮って管理するツール）
  • GitHub = Googleフォトやクラウドストレージ（写真をバックアップして、他の人と共有できるサービス）

タロウ：あー！わかりました！Gitで変更を記録して、GitHubにアップロードするってことですね！

先輩：完璧！さらにGitHubには便利な機能があるんだ：

  • 📤 バックアップ：パソコンが壊れても安心
  • 👥 チームで共有：複数人で同じコードを編集できる
  • 🌐 Webサイトの公開：GitHub Pagesで無料でサイトを公開できる
  • 👀 コードレビュー：他の人に変更をチェックしてもらえる

タロウ：へぇ〜！じゃあGit単体でも使えるけど、GitHubと組み合わせるともっと便利ってことですね！

先輩：その理解でOK！実際、個人で使ってても、GitHubにアップしておけば安心だからね。

第3話：「リポジトリって何？」

タロウ：Gitの説明でよく「リポジトリ」って言葉が出てくるんですけど、これって何ですか？

先輩：リポジトリ（Repository）は、「プロジェクトのフォルダ」だと思えばいいよ。 でも、ただのフォルダじゃなくて、「変更履歴も一緒に保存されてるフォルダ」なんだ。

タロウ：変更履歴も一緒に...？

先輩：そう。例えば、「my-website」っていうフォルダがあるとして：

my-website/
├── index.html
├── style.css
├── script.js
└── .git/  ← ここに変更履歴が保存される！

先輩：この.gitフォルダに、全ての変更履歴が記録されてるんだ。 だから、このフォルダごと「リポジトリ」って呼ぶんだよ。

タロウ：なるほど！じゃあ、普通のフォルダにGitの機能がついたものがリポジトリってことですね！

先輩：完璧！さらに、リポジトリには2種類あるんだ：

種類 場所 説明
ローカルリポジトリ 自分のパソコン 自分だけが見られる、作業用のリポジトリ
リモートリポジトリ GitHub（インターネット上） チームで共有する、公開用のリポジトリ

タロウ：ということは、自分のパソコンで作業して、GitHubにアップロードするってことですか？

先輩：そういうこと！これが基本の流れだよ。

第4話：「コミットって何？セーブとは違うの？」

タロウ：「コミット」っていうのもよく聞くんですけど、これって「保存」のことですよね？

先輩：惜しい！でもちょっと違うんだ。まず整理しよう：

  • 保存（Save） = ファイルの内容をディスクに書き込む（Ctrl+S / Cmd+S）
  • コミット（Commit） = 変更内容を「変更履歴」として記録する

タロウ：えっと...違いがよくわからないです...

先輩：ゲームで例えてみようか！

  • 保存（Save） = ゲームのオートセーブ（常に最新の状態を上書き）
  • コミット（Commit） = セーブポイント（特定のタイミングで記録を残す）

タロウ：あー！セーブポイント！

先輩：そう！ゲームで「ボス戦の前」とか「クリアした後」とかでセーブするでしょ？ コミットも同じで、「意味のある変更をした時」に記録を残すんだ。

タロウ：じゃあ、ファイルを保存（Ctrl+S）するのは別に、コミットっていう操作が必要なんですね？

先輩：その通り！実際の流れはこうなる：

  1. ファイルを編集する（例：index.htmlに新しいセクションを追加）
  2. 保存する（Ctrl+S / Cmd+S）← まだコミットじゃない！
  3. コミットする「新しいセクションを追加しました」というメッセージと一緒に記録

タロウ：なるほど！コミットの時にメッセージを書くんですね！

先輩：そう！このメッセージが超重要。後で見返した時に「何をしたか」がわかるからね。

良いコミットメッセージの例：
  • ✅ 「トップページにお客様の声セクションを追加」
  • ✅ 「問い合わせフォームのバリデーションを修正」
  • ✅ 「レスポンシブデザインに対応」
悪いコミットメッセージの例：
  • ❌ 「修正」
  • ❌ 「update」
  • ❌ 「test」

これだと後で見返した時に何をしたかわからない！

第5話：「ステージング？なんでコミット前にもう一手順あるの？」

タロウ：コミットの前に「git add」っていうコマンドを使うって聞いたんですけど、 なんでコミット前にもう一手順必要なんですか？

先輩：これもみんな最初混乱するところだね！「ステージング」っていう仕組みなんだ。 買い物で例えてみようか。

タロウ：買い物...ですか？

先輩：スーパーで買い物する時を思い出して：

  1. 商品を手に取る（ファイルを編集）
  2. 買い物カゴに入れる（git add = ステージング）← ココ！
  3. レジで会計する（git commit）

タロウ：あー！買い物カゴに入れるのがステージングなんですね！

先輩：そういうこと！なんでこれが必要かというと、「コミットしたいファイルを選べる」からなんだ。

タロウ：えっと...どういうことですか？

先輩：例えば、同時に3つのファイルを編集したとしよう：

  • index.html - トップページの修正（完成した）
  • about.html - 会社概要ページの修正（完成した）
  • test.html - 実験用のファイル（まだ途中）

先輩：この時、完成したindex.htmlとabout.htmlだけをコミットして、 test.htmlはまだコミットしたくないよね？

タロウ：確かに！途中のファイルは記録に残したくないです。

先輩：だからこうする：

git add index.html      # index.htmlをステージング（買い物カゴに入れる）
git add about.html      # about.htmlもステージング
# test.htmlはaddしない = コミットされない
git commit -m "トップページと会社概要ページを更新"

タロウ：なるほど！コミットしたいものだけを選んでステージングするんですね！

先輩：完璧！だから流れはこうなる：

Gitの基本的な流れ： 
1. 【作業ディレクトリ】
   ↓ ファイルを編集・保存
   ↓
2. 【ステージングエリア】
   ↓ git add（コミットするファイルを選ぶ）
   ↓
3. 【ローカルリポジトリ】
   ↓ git commit（変更を記録）
   ↓
4. 【リモートリポジトリ】
   ← git push（GitHubにアップロード）

第6話：「プッシュとプル、なにが起きてるの？」

タロウ：コミットまではわかったんですけど、「プッシュ」と「プル」ってのもよく聞きます。 これは何ですか？

先輩：いい質問！これはローカルリポジトリ（自分のパソコン）とリモートリポジトリ（GitHub）のやり取りだよ。

  • プッシュ（Push） = 自分のパソコン → GitHub（アップロード）
  • プル（Pull） = GitHub → 自分のパソコン（ダウンロード）

タロウ：あ、シンプルですね！

先輩：でもね、実はちょっと深い意味があるんだ。例を見てみよう：

🔼 プッシュの流れ

# 1. ファイルを編集して保存
# 2. ステージングエリアに追加
git add index.html

# 3. ローカルリポジトリにコミット
git commit -m "ヘッダーのデザインを変更"

# 4. GitHubにプッシュ
git push

👆 ここで初めてGitHubに反映される！

タロウ：え、コミットしただけじゃGitHubには反映されないんですか？

先輩：そう！これ超重要！

  • コミット = 自分のパソコン内に記録（まだGitHubには送られてない）
  • プッシュ = GitHubに送る（ここで初めて他の人も見られる）

タロウ：じゃあ、チームで作業してる時は、プッシュしないと他の人に共有されないんですね！

先輩：その通り！逆に、他の人がプッシュした変更を取り込むのが「プル」なんだ。

🔽 プルの流れ

# チームメイトがGitHubにプッシュした変更を取り込む
git pull

👆 GitHubの最新版を自分のパソコンにダウンロード

タロウ：なるほど！じゃあチームで作業する時は：

  1. 作業を始める前にgit pullで最新版を取得
  2. 自分の作業をしてgit commit
  3. 完成したらgit pushで共有

っていう流れですね！

先輩：完璧！本当にその通りだよ！

第7話：「ブランチって何？なんで必要なの？」

タロウ：「ブランチ」っていう概念もあるって聞いたんですけど...これは何ですか？

先輩：ブランチは「並行世界」みたいなものだよ！

タロウ：並行世界...？？

先輩：例えば、今動いてるWebサイトがあるとして、「新しいデザインを試したい」と思ったとする。 でも、本番のサイトをいきなり変更したら、失敗した時に困るよね？

タロウ：確かに！本番サイトが壊れたら大変です...

先輩：だからブランチを使うんだ！

main（本番）
  │
  │  ← ここで分岐！
  ├──→ new-design（実験用）
  │       │
  │       ├── 新しいデザインを試す
  │       ├── 色々実験する
  │       └── 完成したらmainに合流（マージ）
  │
  └──→ mainは安全にそのまま

タロウ：おお！本番を壊さずに実験できるんですね！

先輩：そういうこと！実際の使い方を見てみよう：

# 新しいブランチを作成
git branch new-design

# そのブランチに切り替え
git checkout new-design
# または、作成と切り替えを同時に：
git checkout -b new-design

# ここで自由に実験できる！
# ファイルを編集...
git add .
git commit -m "新しいデザインを実装"

# 完成したらmainブランチに戻る
git checkout main

# new-designブランチの変更をmainに統合（マージ）
git merge new-design

タロウ：なるほど！じゃあ失敗しても、そのブランチを捨てればいいんですね！

先輩：その通り！ブランチのメリットをまとめると：

  • ✅ 安全：本番コードを壊さずに実験できる
  • ✅ 並行作業：複数の機能を同時に開発できる
  • ✅ レビュー：完成してからチームにレビューしてもらえる
  • ✅ ロールバック：失敗したらブランチごと削除すればOK
実際の開発現場では：
  • main（またはmaster）= 本番環境
  • develop = 開発環境
  • feature/new-header = 新機能開発用
  • bugfix/login-error = バグ修正用

こんな感じで、用途ごとにブランチを作るのが一般的だよ！

第8話：「みんながつまずくポイント」

タロウ：だいぶわかってきました！でも、実際使う時に気をつけることってありますか？

先輩：いい質問！みんながよくつまずくポイントをまとめておくね。

❌ よくある間違い1：コミットせずにプッシュしようとする

# ❌ 間違い
git add index.html
git push  ← コミットしてないのにプッシュ！

# ✅ 正しい
git add index.html
git commit -m "ヘッダーを修正"
git push

❌ よくある間違い2：プルせずに作業を始める

先輩：チームで作業してる時、他の人が変更をプッシュしてることがあるんだ。 だから、作業を始める前には必ずgit pullして最新版を取得しよう！

# ✅ 作業開始前の習慣
git pull          # 最新版を取得
# 作業する...
git add .
git commit -m "○○を追加"
git push

❌ よくある間違い3：コミットメッセージが雑

# ❌ 悪い例
git commit -m "update"
git commit -m "修正"
git commit -m "test"

# ✅ 良い例
git commit -m "トップページのヘッダーデザインを修正"
git commit -m "お問い合わせフォームのバリデーションを追加"
git commit -m "レスポンシブ対応のため、CSSを調整"

❌ よくある間違い4：.gitを削除してしまう

先輩：.gitフォルダには全ての変更履歴が入ってるから、これを削除すると全て失われる！ 間違っても削除しないように！

❌ よくある間違い5：パスワードやAPIキーをコミットしてしまう

先輩：これは超危険！GitHubに公開したら、世界中の人が見られるからね。

コミットしてはいけないファイル：
  • ❌ .env - 環境変数（パスワードやAPIキー）
  • ❌ config.secret.json - 秘密情報
  • ❌ node_modules/ - ライブラリ（容量が大きい）

これらは.gitignoreファイルに書いて、除外しよう！

第9話：「実際の作業の流れをまとめると？」

タロウ：色々教えてもらったので、最後に実際の流れをまとめてもらえますか？

先輩：OK！実際のWebサイト制作の流れを見てみよう。

📝 ケース1：個人で新しいプロジェクトを始める

# 1. GitHubでリポジトリを作成（ブラウザで操作）

# 2. 自分のパソコンにクローン（コピー）
git clone https://github.com/username/my-project.git
cd my-project

# 3. ファイルを作成・編集
# index.html, style.cssなどを作成...

# 4. 変更をコミット
git add .
git commit -m "初回コミット：基本的なHTMLとCSSを追加"

# 5. GitHubにプッシュ
git push

📝 ケース2：既存プロジェクトに新機能を追加

# 1. 最新版を取得
git pull

# 2. 新しいブランチを作成
git checkout -b feature/contact-form

# 3. ファイルを編集
# お問い合わせフォームを追加...

# 4. 変更をコミット
git add contact.html
git commit -m "お問い合わせフォームページを追加"

# 5. GitHubにプッシュ
git push -u origin feature/contact-form

# 6. GitHubでプルリクエストを作成（ブラウザで操作）

# 7. レビューを受けて、OKならmainにマージ

📝 ケース3：チームで作業する日常

# 朝、作業を始める前
git pull  # チームメイトの変更を取得

# 自分の作業
# ファイルを編集...
git add .
git commit -m "ヘッダーのナビゲーションを改善"

# 昼休み前にプッシュ（バックアップ）
git push

# 午後、また作業開始
git pull  # 念のため最新版を取得

# 作業...
git add .
git commit -m "フッターにSNSリンクを追加"

# 終業前にプッシュ
git push

タロウ：わかりやすい！こうやって見ると、そんなに難しくないですね！

先輩：そうそう！最初は難しく感じるけど、慣れたらこの流れが自然になるよ。 ポイントは：

  • ✅ こまめにコミット（1機能ごと）
  • ✅ 作業前にgit pull
  • ✅ わかりやすいコミットメッセージ
  • ✅ 実験はブランチで

第10話：「GitHub PagesでWebサイトを公開してみよう！」

タロウ：最後に、GitHubでWebサイトを公開する方法も教えてください！

先輩：おっ、いいね！GitHub Pagesを使えば、静的なWebサイトを無料で公開できるんだ。

🌐 GitHub Pagesでの公開手順

  1. リポジトリを作成
    GitHubで新しいリポジトリを作る（例：my-website）
  2. HTMLファイルを配置
    index.htmlをリポジトリのルートに配置
  3. プッシュ
    GitHubにファイルをプッシュ
  4. GitHub Pagesを有効化
    リポジトリの Settings → Pages → Source を「main」ブランチに設定
  5. 公開完了！
    https://username.github.io/my-website/でアクセスできる！

タロウ：え！こんなに簡単なんですか！

先輩：そう！しかも：

  • ✅ 完全無料
  • ✅ HTTPSで自動的にセキュア
  • ✅ ファイルをプッシュするだけで自動更新
  • ✅ 独自ドメインも設定可能

タロウ：すごい！早速試してみます！

活用例：
  • 個人のポートフォリオサイト
  • プロジェクトのドキュメント
  • 会社の紹介ページ
  • イベントのランディングページ

まとめ：これだけは覚えよう！

🎯 重要な概念

用語 一言で言うと
Git ファイルの変更履歴を管理するツール
GitHub Gitのリポジトリをクラウドで管理するサービス
リポジトリ プロジェクトのフォルダ（変更履歴付き）
コミット 変更を履歴として記録すること
プッシュ ローカル → GitHubにアップロード
プル GitHub → ローカルにダウンロード
ブランチ 並行世界（実験用の分岐）

🔄 基本的な作業の流れ

1. git pull          # 最新版を取得
2. ファイルを編集    # 実際の作業
3. git add .         # ステージング
4. git commit -m "メッセージ"  # コミット
5. git push          # GitHubにアップロード

💡 初心者へのアドバイス

  • ✅ 最初は難しく感じるけど、使えば使うほど慣れる
  • ✅ こまめにコミット・プッシュする習慣をつけよう
  • ✅ コミットメッセージは未来の自分へのメモ
  • ✅ 失敗を恐れずに実験しよう（ブランチを使えば安全）
  • ✅ わからなくなったら、git statusで現在の状態を確認

タロウ：先輩、ありがとうございました！Git/GitHubのことがすごくよくわかりました！

先輩：どういたしまして！あとは実際に使ってみることが一番の勉強だよ。 頑張ってね！
